# Scenario 1:
**Title:** SBP-01: Successful single bet placement with valid selection and stake
**Priority:** Critical
**Risk Rationale:** This is the main user flow. If a valid single bet cannot be placed successfully, the feature fails its core purpose. It also verifies the most important state changes: loading state, successful response, balance deduction and receipt content.
**Steps:**
1. Open the web app as a valid user
2. From the match list select one valid odds button for a match
3. Verify the bet slip shows exactly one active selection
4. Enter a valid stake
5. Click on Place Bet
6. Verify that the loading state of the bet is displayed
7. Wait for placement to complete
8. Verify balance was updated
9. Review the success receipt
10. Close the receipt 
**Expected Results:**
- The selected match is added to the bet slip
- The bet slip shows the active selection, entered stake, available balance and potential payout
- On submit the Place Bet button changes to Placing... and is in loading state
- Placement resolves to success
- Stake is deducted from the user's balance
- Success receipt modal appears showing: 
    - Bet ID
    - Match details
    - Selection
    - Stake
    - Odds at placement
    - Potential payout - correctly calculated
    - Placement timestamp
- Receipt is closed successfully and returns the user to the main flow with no active selections remaining

# Scenario 2:
**Title:** SBP-02: Changing odds selection replaces previous selection and mantains single active bet
**Priority:** High
**Risk Rationale:** The feature explicitly requires one active selection at a time. If the interface allows the user to select multiple odds or fails to replace the old one, users may place incorrect bets or see inconsistent slip data. 
**Steps:**
1. On a match from the match list click odds 1 (home)
2. Verify the bet slip updates
3. On the same match click odds X (draw)
4. Review the bet slip content
5. Select an odds button from another match in the list
6. Verify the bet slip content updates
7. Click on Remove All selections
8. Verify bet slip content
9. Select an odds button from a match
10. Click on the remove x from the selection
11. Verify bet slip content
**Expected Results:**
- First click adds one selection to the bet slip
- Clicking a new odds button (sam match or from another match) replaces the previous selection
- The bet slip always shows only one active selection at a time
- Per-selection remove x, removes the active selection
- Remove All button removes the selection from the bet slip

# Scenario 3:
**Title:** SBP-03: Insufficient balance prevents bet placement
**Priority:** High
**Risk Rationale:** A defect here could allow users to over-bet, creating accounting inconsistencies or UI/API mismatch.
**Steps:**
1. Observe the available balance in EUR
2. Select a valid match outcome
3. Enter a stake greater than the available displayed balance
4. Click Place Bet
**Expected Results:**
- System reject operation
- UI error message is displayed: "Insufficient balance"
- No success receipt is displayed
- Stake is not deducted from balance

# Scenario 4:
**Title:** SBP-04: Stake validation - is required and numeric
**Priority:** Critical
**Risk Rationale:** Invalid stake handling is high-risk validation area because it's enforced by the requierement in the UI and in API. Missing or non-numeric stake values can cause invalid placement attempts and inconsitent error handling
**Steps:**
1. Select a valid match outcome
2. Leave the stake input field empty
3. Click Place Bet button
4. Enter a non-numeric value (like: abc!#$)
5. Click on Place Bet button
**Expected Results:**
- With no stake entered the placement is blocked/rejected
- Clear validation feedback is shown
- With non-numeric stake entered, the placement is rejected
- No bet is placed in either case
- No receipt is shown
- Balance remains unchanged

# Scenario 5:
**Title:** SBP-05: Stake validation - minimum & maximum value, decimal precision
**Priority:** Critical,
**Risk Rationale:** Boundary defects are common and high impact in money-related features. The rules for the stake are strict
**Steps:**
1. Select a valid match outcome
2. Enter stake 0.99
3. Attempt to place the bet
4. Enter stake 1.00
5. Attempt to place the bet
6. Enter stake 100.00
7. Attempt to place the bet
8. Enter stake 100.01
9. Attempt to place the bet
10. Enter stake 10.001
11. Attempt to place the bet
**Expected Results:**
- 0.99 is rejected with minimum stake validation feedback
- 1.00 is accepted as the minimum valid stake
- 100.00 is accepted as the maximum valid stake
- 100.01 is rejected with maximum stake validation feedback
- 10.001 is rejected for exceeding 2 decimal places
- Invelid values do not place a bet or change the balance
- Valid values proceed normally to placement

# Scenario 6:
**Title:** Placement failure shows error modal, Rebet retries, Close clears selection/stake
**Priority:** High
**Risk Rationale:** Failure-handling quality is important for trust and recovery.
**Steps:**
1. Select a valid match outcome
2. Enter a valid stake
3. Trigger a placement failure condition (simulate server failure or controlled backend response)
4. Click Place Bet
5. VErify error modal content
6. Click Rebet
7. If failure is resolved, allow retry to complete
8. Repeate failure flow and click Close
9. Repeate failure flow and click top-right X
**Expected Results:**
- On submit the button enters Placing... state
- The flow resolves to a final failure outcome and is not stuck in loading state
- An error modal appears with title Something went wrong and explanatory text
- Rebet closes the modal and retries placement
- Close clears current selection and stake and user returns to main flow
- Top-rigt X clears current selection and stake and user returns to main flow
